Аналіз архітектурних рішень (Лабораторна робота №2)
1. Порівняння з Лабораторною роботою №1

У першій лабораторній роботі застосовувався класичний "Transaction Script" підхід:

    Структура: Весь код був зосереджений у трьох класах (Controller, Service, Repository/Entity).

    Залежності: Бізнес-логіка була жорстко прив'язана до анотацій Hibernate (@Entity) та фреймворку Spring.

    Тестування: Було неможливо протестувати логіку без підняття контексту бази даних.

У другій лабораторній роботі проект рефакторено згідно з принципами Clean Architecture (Чистої архітектури). Тепер проект розділений на 4 незалежні шари: Domain, Application, Infrastructure, Presentation.
2. Переваги розділення на шари

    Незалежність від БД: Шар Domain не знає про PostgreSQL. Це дозволяє змінити систему зберігання даних (наприклад, на MongoDB або In-memory) без зміни бізнес-правил.

    Тестованість: Доменна логіка тепер покривається швидкими Unit-тестами, які запускаються за мілісекунди без Docker чи Spring.

    Дотримання DIP: Використання інтерфейсів (портів) у домені дозволяє інвертувати залежності: тепер інфраструктура залежить від домену, а не навпаки.

3. Недоліки та ускладнення

    Надмірність коду (Boilerplate): Поява маперів (TaskMapper) та розділення моделей на Task та TaskEntity збільшили обсяг коду.

    Складність навігації: Код рознесений по різних пакетах, що вимагає більше часу на пошук потрібного класу.

    Mapping Overhead: Необхідність постійного перетворення об'єктів між шарами створює невелике навантаження на CPU.

4. Обґрунтування вибору моделі: Rich Domain Model

Для реалізації домену було обрано Rich Domain Model (Багата модель домену).

    Причина: Це дозволяє інкапсулювати інваріанти (бізнес-правила) безпосередньо в об'єкті Task.

    Приклад: Перевірка на порожнє ім'я або валідація дати відбувається в конструкторі та сетерах класу Task. Це гарантує, що в системі не може існувати "некоректного" об'єкта, незалежно від того, який сервіс його створив.

    Порівняння: На відміну від Anemic Model, де об'єкти — це лише набори даних (DTO), Rich Model робить код більш об'єктно-орієнтованим та захищеним від помилок розробника.

5. Domain Factory та Обробка помилок

Впроваджено TaskFactory для централізованого створення об'єктів. Якщо дані некоректні, фабрика або сама модель викидають DomainException. Ця помилка є чистою Java-помилкою, вона не знає про HTTP-статуси, що дозволяє використовувати домен у будь-якому середовищі (наприклад, у консольній програмі або Telegram-боті).
