
1. Зміни у структурі проєкту

У Лабораторній роботі 2 використовувався класичний Layered Architecture (Контролер — Сервіс — Репозиторій). У Лабораторній роботі 3 структура еволюціонувала до CQS (Command Query Separation):

    Розщеплення логіки: Замість одного великого TaskService, з’явилися окремі CommandHandler (для мутацій) та QueryHandler (для читання).

    DTO як стандарт: Вхідні дані тепер чітко розмежовані на CommandDTO та QueryDTO.

    Domain Layer: З’явився виражений шар домену, де зосереджені інтерфейси та бізнес-правила, що робить інфраструктуру (GORM, Gin) лише деталями реалізації.

2. Переваги CQS: Чим допомогло розділення?

Розділення операцій запису та читання принесло наступні переваги:

    Покращена читаність: Кожен обробник (Handler) відповідає лише за одну дію. Файл хендлера не розростається до тисяч рядків.

    Спрощене тестування: Ми отримали можливість писати ізольовані Unit-тести для команд без потреби в базі даних, використовуючи моки.

    Зменшення побічних ефектів: Читання даних (Query) гарантовано не змінює стан системи, що робить програму більш передбачуваною.

3. Недоліки CQS: Що ускладнилося?

    Бойлерплейт (Надмірний код): Кількість файлів та структур зросла. Для простого CRUD-додатка створення окремих хендлерів та DTO для кожної дії виглядає як "overengineering".

    Складність навігації: Тепер логіка створення завдання розпорошена між контролером, DTO, хендлером та репозиторієм, що вимагає більше часу на відстеження ланцюжка викликів.

4. Command/Query Handler vs Monolithic Service

У підході "Сервіс, який робить все сам", один клас або структура має забагато причин для зміни (порушення Single Responsibility Principle).

    Service: Якщо нам потрібно змінити логіку валідації при створенні та логіку фільтрації при пошуку — ми редагуємо один і той самий файл.

    CQS Handlers: Логіка запису та читання фізично розділена. Це запобігає ситуаціям, коли зміна методу Update випадково ламає метод Get.

5. Вплив на розширюваність

CQS робить систему надзвичайно гнучкою для масштабування:

    Нові команди: Додавання нового функціоналу (наприклад, "Архівувати завдання") відбувається через створення нового обробника. Це не зачіпає існуючий код (Open-Closed Principle).

    Масштабування читання: Якщо з часом навантаження на читання зросте, ми можемо оптимізувати лише QueryHandler (наприклад, додати кешування або використовувати іншу БД для запитів), не змінюючи складну логіку запису.

6. Різниця між Query Data та Domain Model

У нашій реалізації Query може повертати дані, що відрізняються від model.Task.

    Чому це важливо? Доменна модель містить технічні поля (ID, CreatedAt, DeletedAt, паролі або внутрішні статуси). Для API запитів (Query) ці дані часто не потрібні або мають бути агреговані.

    Безпека та гнучкість: Повертаючи спеціальні структури (Projections), ми не виставляємо нутрощі бази даних назовні. Це дозволяє змінювати структуру таблиць у БД, не ламаючи контракт із фронтендом.

Висновок

Перехід до CQS у Лабораторній роботі 3 значно підвищив рівень контролю над кодом та полегшив тестування. Попри збільшення кількості файлів, така архітектура є більш стійкою до змін у великих проєктах, де бізнес-логіка постійно розвиваєтьс
